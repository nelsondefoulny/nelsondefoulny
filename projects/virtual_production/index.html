    Virtual Production

    Introduction

    As the technical director of Blue Node I was leading a small developer team. Over the year we implemented a lot of tools to push further our Virtual Production setups.

    Virtual Production have many different meaning depending on the industry and the context. The most generic definition is to consider any production that leverage real time technology to replace classic offline rendering in order to provide new options. At Blue Node each production had its own workflow and tools depending on the needs. Therefore I had the chance to work with a wide range of technologies and devices.

    Before entering in too much detail I would like to show some of the production we have done.

    Virtual Production Core Concepts

    Explaining virtual production in depth would take a dedicated book so I will just provides an overview.

    The main idea behind these productions is to get physical camera location in real time (Tracking). Using this information we can compute the 3D camera frustum and render a virtual scene. This gives us an image with correct perspective from camera view point. Depending on the context we can use the created image to enhance the physical camera captured image.

    In case of Augmented Reality (AR) the rendered image is just added in front of the captured image. This is used to add virtual objects in front on actors.

    In case of In Camera VFX (ICVFX) the goal is to send this image in a physical LED wall behind the actors. This creates the same effect as a traditional Green Screen but the result is visible immediately instead of waiting post production. The second advantage is that the lights match the 3D environment which is especially useful to create an ambiance or to handle reflective objects. On top of the camera tracking this technic also requires a 3D Mesh that match the physical LED screen in order to re-project renderer image.

    One of the problem with ICVFX is that large plans or not really feasible because the camera will capture the surrounding studio which we don't want to see. Ideal solution is to use a bigger LED wall but this is really expensive. The second solution is to create en Extend (XR). The idea is to render the same scene and perspective as the LED wall image and use this as an AR mask outside of the LED.

    All of the production above leverage camera tracking and unreal engine real time rendering. But th actual setup might include one or multiple concepts (ICVFX / AR / XR).

    What it looks like in practice

    A virtual production setup usually includes following elements :
    - One or multiple camera.
    - One tracking system per tracked camera.
    - A genlock to synchronize devices. It is a pulse indicating the beginning or end of each frame.
    - A timecode to get a consistent time source across multiple devices. It can also be use control some playback or trigger events.
    - ICVFX : a LED screen and its processors
    - ICVFX : a cluster of synchronize rendering servers (LED has too many pixel for one machine). The underlying technology is called nDisplay and is developed by Epic Games.
    - AR : one rendering server per video feed (usually one per camera).
    - A operator computer that serves as a hub to control servers.
    - A graphic designer dedicated computer.
    - A bunch of networks using a variety of protocol to transfer data across devices (udp, tcp, dmx, osc, etc).

    This list is just an overview of important devices for the 3D related tasks. There are a lot of other devices to control other aspects of the production (lights, recording, audio, set, etc).

    Personal Contribution

    I want to present a short list of the tools my team and I developed. A big part of the job was linked to the specifications of each production and we produced a lot of one shot code that is not worth mentioning in detail.

    Checks

    One of the principal difficulty in Virtual Production is that it is a long chain of devices and software and the only return is the final image. Identifying the source of a problem or even noticing an existing problem is not trivial. With my team we developed multiple tools (bash, python, unreal, etc) meant to helped us with diagnosis.

    Source Control

    In order to sync rendering servers we extend Epic Games switchboard to support Plastic SCM. We also add some real time diagnosis of the machine performance and potential divergence with other servers.

    Operator Control

    This is the main Unreal Engine tool we develop. It is more a suit of tools that was design to setup and operate the system. Overall features includes :
    - Create/Load per machine configuration.
    - Load/Save/Sync unreal engine maps.
    - Control and Diagnose machine performances in real time.
    - Centralize message and error logs from all machines.
    - Trigger events and sequences.
    - Complex compositing setup for ICVFX inner and outer frustum.
    - Handle network communication with third party devices using OSC, UDP and DMX protocols.

    LED Wall Creator

    This tool was made to create LED mesh for ICVFX setups. This was motivated by the difficulty we faced to create this mesh. In order to have valid projection we need a really accurate mesh and all other technics (measure, lasers, lidar, etc) we tried failed to provide a quick and repeatable creation process.

    In the first version of the tool the mesh was built vertically on top of the spline. Spline point were projected in a pixel accurate way into the LED floor. Therefore moving spline points was reflected in real time into the physical floor which allow to create the curved LED Wall.

    For this tool i developed :
    - 3D to 2D location conversion
    - Dynamic mesh creation, UV editing and static mesh baking from Unreal Engine Editor
    - Automatic reload of nDisplay mesh at runtime (in servers)
    - Edit propagation through Unreal Engine Multi-User tool

    Technical Mire

    I have developed a technical Mire for LED Screens. This tool has multiple motivation but the main idea was to validate LED screen setup. Among other it helped a lot to validate LED screen layout and color as well as nDisplay LED mapping. It often happened that some tile of the LED was inverted either in the LED processor mapping or physically. The same goes for colorimetry that might be off for multiple reason. Especially it makes it visually obvious if the LED is displaying 8 bit rather than 10 bit colors. In term of development it is just a shader but getting pixel accurate precision wasn't that simple.

    Scene Snapshot

    This tool was built on top of multiple unreal plugins and was developed to save and organize scene snapshots and replay them on demands. Handling multiple scene versions in Unreal Engine can be quite challenging especially because of the fast paced production we were working on. Duplicating a map and using source control was definitely not an option as it would have taken too much time on top of forcing the rendering cluster reboot. Therefore we built the tool on top of Epic Games Level Snapshot tool. On top of it we developed a complex filtering setup tied to a sublevel system that hold graphical content leaving the persistent level free to contain technical stuff. We also developed multiple snapshot reloading methods (code, sequence, custom UI). The most complex part was make sure that the snapshot reloads properly on nDisplay Cluster as the Level Snapshot is an Editor tool running on the operator machine while the server are in runtime mode.

    Multi Unreal Per Server

    For Hondelatte Raconte season 2 we wanted to swap scene in real time to create a theater effect. Normally in ICVFX production we use the first GPU to render camera frustum and the second GPU to render a 360 image of the scene for the outer frustum which serve as ambient light. This is done using one unreal engine instance per machine leveraging nDisplay multi GPU support.

    For this production we decided to use pre rendered images for the outer frustum which free a GPU. I extended Unreal Engine TextureShare plugin in order to support texture transfer between two unreal instances running on the same machine. This allow me to setup two Unreal Instance per server, one on each GPU, each unreal rendering its own scene. I then leverage the texture share across GPU to composite final image in the first GPU (the one with display port output).

    Modifying texture share was not trivial but the real complexity was not coming from this part of the plan. The principal problem was to add control for loading not one but two scene per machine and dispatch them properly on both instances. None of Unreal Engine plugins dedicated to virtual production supports such setup. Therefore we had to extend operator control tool, compositing setup, outer frustum images, multi user behavior, error management, and basically all of the existing tools we were using.

    This was a big challenge but during the production it worked well and even goes beyond expectation by lowering the amount of time lost in between scenes thanks to being able to preload scene in advance.

    Graphics Designers Support

    Another important part of my job was to support graphics designers. I have developed multiple small tools related to content creation like a spline placement tool, a conveyor, some rendering related scripts, and of course some turnkey solutions to integrate their work in the context of virtual production.
